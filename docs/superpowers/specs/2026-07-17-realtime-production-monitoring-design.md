# 실시간 생산 감시 — 설계 문서

- 작성일: 2026-07-17
- 상태: 설계 승인됨 (구현 계획 미작성)
- 관련: `production_records`, `downtime_entries`, `machine_logs`, `production_shift_states`

---

## 1. 배경

생산 수량을 **생산이 모두 끝난 다음 날** 일괄 입력하고 있다. 그래서 OEE 는 항상 하루 늦게
나오고, 교대가 끝난 뒤에야 문제를 안다. 실시간 감시가 불가능하다.

문제의 규모는 측정된다 (2026-07-17 운영 DB):

```
production_shift_states   (2026-07-13 ~ 07-16)
  WORKING  4,550건
  MISSING  1,512건   ← 전체의 약 25%
  OFF·HOLIDAY: 0건 (모델엔 있으나 미사용)
```

**4교대 중 1교대가 기록 누락 상태다.** 게다가 그 `MISSING` 안에는 두 가지가 뒤섞여 있다:

- 정말 입력을 잊은 교대 (찾아서 채워야 함)
- 도색·정비로 애초에 안 돌린 교대 (정상)

**이 둘을 구분할 수단이 없다.** 그래서 "800대 중 누락이 없어야 한다"는 명제를 지킬 수 없다.

### 이미 있는 것과 없는 것

조사 결과, 실시간 OEE 의 3요소 중 **가동률은 이미 계산 가능하다**:

| 테이블 | 현황 | 시각 정보 |
|---|---|---|
| `downtime_entries` | 165건, `end_time` NULL 허용 | **있음** (`start_time`/`end_time`) |
| `production_records` | 교대당 1행 집계 | **없음** (`output_qty` 는 그냥 정수) |

빠진 조각은 **생산 수량의 시계열** 하나다.

### 장기 비가동은 집이 없다

```
downtime_entries        진행중(end_time NULL): 0건 · 최장 0.50일 · max 720분(=1교대)
                        사유: endmillChange 134 · equipmentFailure 29 · productionModelChange 2

machine_logs            PLANNED_STOP 12건 (1건 열림, 최장 3.02일)
                        NORMAL_OPERATION 115건 (41건 열림, 최장 3.15일)
```

`downtime_entries` 는 스키마상 "진행 중"을 지원하지만 **한 번도 쓰인 적이 없고**, 실제로는
한 교대(720분) 안에 갇혀 있다. 도색 같은 사유도 없다.

반면 `machine_logs` 는 **이미 여러 날에 걸친 상태**를 다룬다(PLANNED_STOP 3.02일, 현재도
1건 열림). 도색은 여기 산다.

**문제는 이 둘이 끊어져 있다는 것이다.** 설비가 `machine_logs` 상 3일째 PLANNED_STOP 인데,
그 3일의 교대들은 OEE 쪽에서 그냥 `MISSING` 으로 남는다.

## 2. 목표 / 비목표

### 목표

1. 교대 중 생산 수량·비가동을 입력해가며 **실시간 감시**
2. 교대 시간과 하루 변경 시간을 넘어 연속적으로 동작
3. **누락된 설비·날짜·교대를 UI 에서 즉시 식별**해 마무리할 수 있게 함
4. 장기 비가동을 작업자가 **시각적으로 인지** ("현재까지 비가동 중입니다")

### 비목표 (명시적으로 범위 밖)

- **설비 자동 연동(PLC/통신)** — 사람이 입력한다
- **QC 별도 역할·화면** — 불량은 다음날 기존 화면에서
- **알림·임계값·에스컬레이션** — 이상 신호는 불필요. 작업자가 **인지**만 하면 된다
- **오프라인 큐잉** — 누적 방식이 흡수한다 (§7 참조)
- **CAPA 변경** — 불변 제약
- **과거 1,512건 MISSING 소급 정리** — 이 기능은 *앞으로*의 누락을 막는다

## 3. 핵심 설계 의도

> **성격이 다른 데이터는 다른 그릇에 담는다.**

진행 중 데이터(진행 보고·비가동)와 확정 데이터(교대 최종 스냅샷)를 **물리적으로 다른 테이블**에
둔다. 그러면 둘을 섞는 사고가 *일어날 수 없다*.

이 원칙은 임의로 고른 것이 아니다. 2026-07-16~17 에 고친 버그 5개 중 3개가 "성격이 다른
데이터를 같은 그릇에 담아서" 생겼다:

- 진행 중 교대(2시간, 60%)와 완료 교대(12시간, 96%)를 같이 평균내면 무의미한 숫자가 나온다
- 미보고(NULL)를 0 으로 뭉개면 멀쩡한 설비가 빨간 0.0% 로 보인다

> **모르는 것을 숫자로 단정하지 않는다.**

불량은 나중에 검사한다 → 교대 중 **품질은 모른다** → **OEE 를 계산할 수 없다**.
품질을 100% 로 가정해 OEE 를 띄우면 방금 고친 버그를 새 기능으로 다시 만드는 것이고,
게다가 항상 낙관적으로 틀린다.

대신 **가동×성능은 완전한 확정값**이다(둘 다 검사와 무관). 현장이 실시간으로 알고 싶은
것도 사실 이것이다 — *설비가 지금 잘 돌고 있나*.

따라서 이 기능의 정확한 이름은 "실시간 OEE"가 아니라
**"실시간 가동×성능 + 검사 후 OEE 확정"** 이다.

## 4. 확정된 요구사항

| 항목 | 결정 | 근거 |
|---|---|---|
| 실시간 지표 | **가동×성능**. OEE 는 `—` | 검사 전 품질은 모름 |
| 진척 표시 | CAPA 대비 누적 + 경과율 | "지금 효율"과 "목표 달성 가능성"은 다른 질문 |
| 입력 값 | **이 교대에서 지금까지 만든 총 개수** (누적) | 입력을 빠뜨려도 합계가 안 틀림 |
| 수량 파악 방법 | **작업자 재량** (시스템이 규정하지 않음) | 카운터 리셋 여부와 무관하게 성립 |
| 비가동 | `downtime_entries` (기존) | 이미 이벤트 기반 |
| 불량 | **다음날 입력** (현행 유지) | 검사가 나중 |
| 장기 비가동 OEE | **계획에서 제외** (집계 미포함) | 표준 OEE 정의. 정비 잘한 설비가 벌받지 않음 |
| 비가동 중 입력 | **잠금.** 정상 전환 후 입력 | 안 도는 설비에 생산량을 넣을 수 없음 |
| 이상 신호 | **없음** | 인지만 하면 됨 |
| CAPA | 변경 없음 | 제약 |

## 5. 데이터 모델

### 5.1 신규 — `production_progress_reports` (append-only)

```
id                uuid        PK
machine_id        uuid        NOT NULL
date              date        NOT NULL   -- 교대 귀속일 (B교대는 시작일)
shift             text        NOT NULL   -- 'A' | 'B'
reported_at       timestamptz NOT NULL   -- 보고 시각
shift_output_qty  integer     NOT NULL   -- 이 교대에서 지금까지 만든 총 개수 (누적)
operator_id       uuid
created_at        timestamptz NOT NULL
```

**필드의 의미는 하나다: "이 교대에서 지금까지 만든 총 개수".**

**작업자가 그 숫자를 어떻게 얻는지는 규정하지 않는다.** 교대마다 리셋되는 카운터를 그대로
읽든, 리셋되지 않는 카운터에서 교대 시작값을 빼든, 완성품을 직접 세든 — 시스템에는
`shift_output_qty` 라는 하나의 의미로 들어온다.

이 구분이 중요하다. 필드를 *메커니즘*("카운터 판독값")으로 정의하면 카운터가 리셋되지 않는
설비에서 규칙이 깨지고, 800대에 유형이 섞이면 계약이 무너진다. *의미*로 정의하면 그 위험이
사라진다. (필드는 하나의 의미를 가져야 하고, 그 의미가 구현 방식에 묶이면 안 된다.)

**append-only.** 오타는 수정이 아니라 새 보고를 추가해 덮는다. "13:00 에 150개였다"는
불변 사실이고, 사실은 고쳐 쓰는 게 아니라 다음 사실로 갱신한다. 스냅샷을 나중에 덮어쓰면
과거가 조용히 바뀐다.

교대 생산량 = **그 교대의 마지막 `shift_output_qty`** (이미 교대 누적이므로 뺄셈 불필요).

### 5.2 역할 분리

| 테이블 | 담는 것 | 성격 | 변경 |
|---|---|---|---|
| `production_progress_reports` | 교대 중 진행 보고 | 진행 중 · 불변 이벤트 | **신규** |
| `downtime_entries` | 비가동 | 진행 중 · 이벤트 | 없음 |
| `machine_logs` | 설비 상태 (여러 날 가능) | 진행 중 · 이벤트 | 없음 |
| `production_records` | **교대 최종 확정값** | 완료됨 · 스냅샷 | **없음** |

`production_records` 를 손대지 않는 것이 이 설계의 핵심이다. 모든 분석 RPC·리포트가 이
테이블을 읽으므로, 의미가 그대로면 **기존 동작이 한 줄도 안 바뀐다.**

### 5.3 누락과 계획 정지의 구분 — 저장하지 않고 파생한다

누락 방지의 심장. 지금 `MISSING` 1,512건에는 "잊음"과 "도색 중"이 뒤섞여 있다. 이를 가른다.

**결정: `production_shift_states` 에 새 status 를 저장하지 않는다.** 대신 조회 시점에
`machine_logs` 를 조인해 분류한다.

| 분류 | 판정식 | `production_records` | OEE 집계 |
|---|---|---|---|
| `WORKING` | `production_records` 존재 | 있음 | 포함 |
| **계획 정지** | 그 교대 구간에 `machine_logs.state='PLANNED_STOP'` 이 걸쳐 있음 | 없어도 정상 | **제외** |
| **누락** | 위 둘 다 아닌데 `MISSING` | 없음 (문제) | 제외 |

**왜 저장하지 않는가.** 열린 로그(`end_time IS NULL`)의 구간은 **시간이 흐르면 늘어난다**
— 3일째 도색이 내일이면 4일째다. 저장하면 이 값을 계속 갱신할 작업(job/trigger)이 필요하고,
그 작업이 밀리면 화면이 거짓을 말한다. 파생하면 **항상 현재 사실과 일치**한다.

트레이드오프: 나중에 로그를 수정하면 과거 분류도 함께 바뀐다. 이는 의도한 것이다 —
잘못 기록한 상태를 고치면 그 결과도 고쳐지는 편이 옳다. (`production_records` 의 스냅샷
불변성과는 성격이 다르다. 저쪽은 *확정된 실적*, 이쪽은 *상태에 대한 해석*이다.)

**집계 제외에 별도 로직이 필요 없다.** `production_records` 행을 만들지 않으면 RPC 모집단
에서 자연히 빠진다. 이미 그렇게 동작한다. 필요한 건 "왜 없는가"를 구분하는 것뿐이고,
그 구분은 화면에서만 쓰인다.

## 6. 계산 규칙

### 6.1 실시간 (교대 중, 주간조 10:00 예시 — CNC-001, tact 72초/개)

교대 상수 (운영 `system_settings` 실측): 가동 720분, 휴식 **110분** → 계획 **610분**.
CAPA = 610 / (72/60) = **508개**.

```
경과 시간     = 08:00 → 10:00                              = 120분
지나간 휴식   = 09:50~10:00 (10분) 만 경과                  =  10분
경과 계획시간 = 120 − 10                                    = 110분
비가동        = downtime_entries 합 (열린 건은 now() 까지)  =   0분
실제 가동시간 = 경과 계획시간 − 비가동                      = 110분
이상 가동시간 = 마지막 보고 60개 × 72초 / 60                =  72분

가동률   = 110 / 110 = 100%     ← 확정
성능률   =  72 / 110 = 65.5%    ← 확정
품질률   =     —                 ← 미확정 (검사 전)
OEE     =     —                 ← 계산 불가
가동×성능 =       65.5%          ← 실시간 신호

진척 = 60 / 508개 (11.8%)  ·  경과 110 / 610분 (18.0%)
       → 진척이 경과보다 느림 = 목표 미달 예상
```

기존 `OEECalculator` 공식을 그대로 쓴다. 분모만 610분이 아니라 *경과 계획시간*일 뿐,
새 계산법을 만들지 않는다. tact 는 **개당** 이며 `cavity_count` 로 나누지 않는다.

### 6.3 휴식 시간대 — 새 설정, 기존 총량과 반드시 일치

`system_settings(category='shift')` 는 총량(`break_time_minutes: 110`)만 알고 **언제인지는
모른다.** 경과 계획시간을 계산하려면 시각이 필요하므로 시간대를 추가한다.

| 교대 | 휴식 시간대 | 합 |
|---|---|---|
| A (08:00–20:00) | 09:50–10:00, 11:20–12:20, 14:50–15:00, 17:30–18:00 | 110분 |
| B (20:00–08:00) | 21:50–22:00, **23:20–00:20**, 02:50–03:00, 05:30–06:00 | 110분 |

B 교대의 `23:20–00:20` 은 **자정을 넘는다.**

**같은 사실의 출처가 둘이 되는 것을 경계한다.** 시간대 합계와 `break_time_minutes` 가
어긋나면 실시간 화면과 확정 OEE 가 서로 다른 말을 한다. 그래서:

- 기존 `plannedRuntime.ts` 는 **건드리지 않는다** (`break_time_minutes` 를 계속 사용).
  기존 저장 경로의 동작이 변하지 않는다.
- 실시간 계산만 시간대를 읽는다.
- **`sum(시간대) === break_time_minutes` 를 테스트로 고정**해 어긋남이 조용히 지나가지
  못하게 한다.

중식·석식 시각이 가끔 바뀌어도 시간대는 **고정값으로 둔다**(현장 결정). 4개 구간이 모두
교대 안에 있으므로 교대가 끝나면 어차피 110분이 다 지나가 계획시간은 610분이 된다 —
**배치는 교대 *중* 정밀도에만 영향을 주고 확정값은 바꾸지 않는다.**

### 6.2 확정 (다음날)

불량 수량 입력 → `production_records` 생성 (지금과 동일한 모양, 생산량은 마지막 보고값
자동 채움) → 품질·OEE 확정. 교대 끝에는 실시간 값과 확정 값이 자연히 수렴한다.

## 7. 에러 처리 — "묻고, 추측하지 않는다"

| 상황 | 처리 |
|---|---|
| **보고값 감소** (150 → 60) | 저장 거부하고 되묻는다. 오타이거나 예기치 못한 상황이다. 조용히 받으면 120개가 증발한다. 값의 의미가 "교대 누적"이므로 감소는 불가능하다 — 파악 방법과 무관하게 성립하는 규칙이다. |
| **비가동 중 입력 시도** | 입력란 잠금 + "정상 가동으로 전환 후 입력하세요" |
| **교대 경계** (B교대 02:00 보고) | `date` 는 **전날**, `shift='B'`. 기존 `shiftUtils` 규칙 그대로. |
| **비가동이 교대를 넘김** (도색) | 다음 교대도 열린 로그를 상속 → 계속 "비가동 중" |
| **두 사람이 같은 설비 입력** | append-only 라 충돌 없음. 최신 보고가 현재값. |

### 오프라인 큐잉이 불필요한 이유

누적 방식을 고른 덕분이다. 10:00 에 통신이 끊겨 못 보내면 11:00 에 그때 값(90)을 보내면
된다. **합계는 여전히 정확**하고 시간 해상도만 잃는다. 증분 방식이었다면 그 1시간이 영영
사라져 오프라인 큐가 필수였을 것이다.

**설계 결정 하나가 기능 하나를 통째로 없앴다.**

## 8. 화면

### 8.1 작업자 태블릿 (기존 `OperatorDashboard` 확장)

```
CNC-001  🟢 가동중    508/508개  가동×성능 96%   [생산량 입력]
CNC-002  ⚠️ 보고없음  마지막 보고 2시간 전         [생산량 입력]
CNC-003  🔴 비가동    3일째 · 도색                [정상가동 전환]
         └ "2026-07-14 09:00부터 현재까지 비가동 중입니다"
CNC-004  📋 마감대기  7/16 야간조 불량 미입력      [불량 입력]
```

비가동 중이면 생산량 입력란을 잠그고, 경과 일수와 사유를 크게 보여주며 전환 버튼만 남긴다.

### 8.2 미완료 상태 3종

| 상태 | 판정 (임계값 없음) | 조치 |
|---|---|---|
| **보고 없음** | 진행 중인 교대 + 진행 보고 0건 | 작업자가 생산량 입력 |
| **마감 대기** | 교대 종료 + 진행 보고 있음 + `production_records` 없음 | 다음날 불량 입력 |
| **비가동 중** | `machine_logs` 열림 (`end_time IS NULL`) | 끝나면 정상 전환 |

**임계값을 도입하지 않는다.** "몇 분 이상 미입력이면 경고" 같은 규칙은 만들지 않는다
(이상 신호 불필요 — §2 비목표). 대신 **사실만 표시**한다:

- `마지막 보고: 2시간 전` — 경과 시간은 사실이지 경고가 아니다
- `3일째 비가동 중` — 일수는 사실이지 이상 신호가 아니다

판단은 작업자가 한다. 시스템은 인지를 돕는다.

**세 번째가 첫 번째를 가려준다.** 도색 중인 설비는 "보고 없음"이 아니라 "비가동 중"이다.
지금 `MISSING` 1,512건이 뒤섞여 있는 이유가 이 구분의 부재다.

### 8.3 관리자·엔지니어

```
미완료 작업                    설비   교대      ← 숫자는 형식 예시 (실측 아님)
  보고 없음 (진행 중)           12     12
  마감 대기 (불량 미입력)       38     47      ← 클릭하면 목록
  비가동 중                      4      —
```

이 화면은 사실 기존 자산의 확장이다. `reporting_coverage` 가 이미 "보고됨/미보고/무효"를
세고, `production_shift_states` 에 `MISSING` 이 이미 있다. **없던 건 사람이 볼 수 있는
창이었다.**

그래서 "800대 중 누락이 없어야 한다"가 **측정 가능한 숫자**가 된다 — `MISSING` 을 0 으로
만드는 것이 목표이고, 현재는 1,512 다.

## 9. 테스트

| 대상 | 방식 |
|---|---|
| 경과 시간 기반 계산 | **`Date` 고정 단위 테스트.** 시각 의존은 flaky 의 원흉 |
| B교대 자정 넘김 | 02:00 보고가 전날 B교대에 귀속되는지 |
| 보고값 감소 | 거부되는지 (변이 테스트로 검증) |
| 미완료 판정 | 도색 중 설비가 "보고 없음"으로 안 잡히는지 |
| **품질 미확정** | OEE 가 `—` 이고 **0% 가 아닌지** ← 재발 방지 |
| 권한 | 작업자가 담당 아닌 설비에 보고를 넣을 수 없는지 |

## 10. 롤아웃

두 덩어리로 나눈다. 각각 그 자체로 동작하는 소프트웨어를 만든다.

### A. 실시간 감시

1. 신규 테이블 `production_progress_reports` + RLS (작업자는 담당 설비만 — 기존
   `assertMachineAccess` 사용)
2. 실시간 계산 (경과 계획시간 기반 가동×성능, CAPA 대비 진척) — §6.1
3. 태블릿 입력·표시 (기존 `OperatorDashboard` 확장). 비가동 중 입력 잠금 — §8.1

→ 완료 시: 작업자가 교대 중 생산량을 입력하고 실시간 가동×성능을 본다.

### B. 누락 방지

4. 미완료 3종 분류 (보고 없음 / 마감 대기 / 비가동 중) — §8.2.
   계획 정지 판정은 **조회 시 `machine_logs` 조인으로 파생**한다. §5.3 대로
   `production_shift_states` 에 새 status 를 저장하지 않으므로 **스키마 변경이 없다.**
5. 작업자용 배지 + 관리자용 요약·드릴다운 — §8.3
6. 다음날 입력 화면에 생산량 자동 채움 (마지막 보고값) — §6.2

→ 완료 시: 누락된 설비·날짜·교대가 보이고 찾아서 마무리할 수 있다.

**기존 동작은 A~B 내내 깨지지 않는다.** `production_records` 를 건드리지 않으므로 이 기능을
쓰지 않는 설비는 지금과 똑같이 동작한다. 단계별로 일부 라인에서만 시범 운영할 수 있다.

B 는 A 의 테이블에 의존하므로 A 를 먼저 한다.

## 11. 가정

- **작업자는 "이 교대에서 지금까지 만든 개수"를 알 수 있다** (현장 확인됨). 방법은 설비마다
  달라도 무방하다 — 리셋되는 카운터를 읽든, 리셋되지 않는 카운터에서 빼든, 직접 세든.
  시스템은 방법을 규정하지 않고 결과값만 받는다.
- 도색 등 장기 정지는 `machine_logs` 의 `PLANNED_STOP` 으로 기록된다. 현재 12건 중 1건이
  열려 있어 이 경로가 실제로 쓰이고 있음이 확인된다.

### 검증된 가정이 아닌 것

- **뺄셈이나 수기 집계의 정확도**는 작업자 공정에 달려 있고 시스템이 보증하지 않는다.
  다만 §7 의 감소 감지가 명백한 오류(150 → 60)는 잡아낸다. 이 규칙은 값의 *의미*가
  "교대 누적"이기 때문에 성립하며, 파악 방법과 무관하다.
