# 통합 입력 콘솔 (운영자) — 설계

작성일: 2026-07-18 · 브랜치: design/feature-prd · 상태: 설계 확정(구현 계획 대기)

## 1. 배경 / 문제

운영자의 데이터 입력이 **3개 진입점**으로 흩어져 있고, 실제 저장 목적지는 **4개**이며 그중 하나는 **중복**이다.

| 진입점 | 저장 | 성격 |
|---|---|---|
| ① 데이터 입력 페이지 | `production_records`(output/defect) + `downtime_entries` | 확정 실적 + 비가동 |
| ② 운영자 화면 · 상태 변경 | `machine_logs` | 설비 상태 |
| ② 운영자 화면 · 생산 실적 입력 | `production_records`(output/defect) | 확정 실적 (**①과 중복**) |
| ③ 운영자 화면 · 진행 보고 | `production_progress_reports` | 교대 중 누적 진척 |

문제: (1) 확정 실적을 ①·②에서 둘 다 넣을 수 있어 충돌·누락의 근원, (2) 성격이 다른 4종 데이터가 3화면에 흩어져 운영자가 매번 "무엇을 어디에" 판단해야 함.

### 현장 현실 (설계를 강제하는 제약)

- **설비가 카운트를 자동으로 제공하지 못한다.** 생산 수량은 전량 수기.
- 교대 교체 시 설비를 **세우지 않고** reset 후 즉시 재가동한다. 그래서 교대 임박 시 작업자가 카운터를 **종이에 적었다가 나중에 옮겨 적는** 상황이 생긴다.
- 따라서 **교대 시간이 지난 뒤에도 그 교대의 최종 수량을 입력할 수 있어야 한다.**
- **불량은 다음날 검사 후** 나온다 → 교대 중·교대 종료 시점엔 품질을 모른다(이미 실시간 기능의 전제. `CLAUDE.md`의 "NULL≠0%").

## 2. 목표 / 비목표

**목표**
- 설비 선택 후 **한 콘솔**에서 모든 입력. 태블릿·작업자·터치 우선.
- 입력 경로 단순화 → 누락 감소 → 짧은 교육으로 현장 적용.
- 업계 권장 패턴 준수: 설비 중심 콘솔 · 발생 순간 5초 입력 · 화면 live 요소 4~5개.

**제약 (workflow 안 깨기)**
- 기존 데이터 종류·시점·저장 구조, 실시간 기능, OEE 계산 규율, 확정 스냅샷을 **그대로 둔다.**
- 모든 변경은 기존을 깨지 않는 **additive**.

**비목표**
- 설비 카운트 자동 연동(현재 불가).
- 실시간 OEE(교대 중 품질 미검사).
- 데이터 입력 페이지 제거 — 관리자 백필용으로 **존치**.

## 3. 결정 요약 (브레인스토밍 Q&A)

| 항목 | 결정 |
|---|---|
| 사용 환경 | 현장 태블릿 · 작업자 (터치 우선, 큰 버튼) |
| 레이아웃 | **A. 마스터–디테일** — 왼쪽 설비 목록 + 오른쪽 콘솔 (현행 OperatorDashboard 진화) |
| 진척 입력 | 콘솔 상주 **인라인** 누적 입력 (카운터 읽어서) |
| output | 진척에서 파생(마지막값). 단 **수동 "교대 마감"**: 진척값 prefill + 원탭 확정, **늦어도 됨** |
| 비가동 | **andon 한 동작**(시작+사유 / 재개) → `machine_logs` + `downtime_entries` 동시 기록 |
| 불량 | **다음날 검사 후**, 작업자 콘솔에서 입력 |
| 미확정 처리 | **X안**: 교대 마감 시 record 생성(불량=NULL, 확정대기). 리포트는 확정대기/확정 구분 |
| 마감/불량 대기 | 입력 전까지 **무기한**, 배지로 독촉. **자동 마감 없음** |
| 귀속 | 늦게 입력해도 그 교대로 귀속 (입력 시각 무관) |
| 데이터 입력 페이지 | 관리자 백필용으로 **존치** |

## 4. 설계

### 4.1 레이아웃 (A · 마스터–디테일)

태블릿 가로. 왼쪽 = 담당 설비 그리드(설비명 + 상태·진척 미리보기, 선택 강조). 오른쪽 = 선택 설비 콘솔. 현행 `OperatorDashboard`를 이어받아, 흩어진 탭/모달(상태변경·생산실적·진행보고)을 **하나의 콘솔 패널로 통합**한다.

### 4.2 콘솔 섹션 (위→아래, live 요소 4~5개 유지)

1. **실시간 지표** — 상태(정상/비가동) · 가동×성능 · 진척/CAPA · 교대 경과율. 품질/OEE는 없음(미검사).
2. **진척 입력** (파랑, 주 동작) — 인라인 큰 숫자 "이 교대 누적 생산량" + 저장. "교대 종료 시 이 값이 확정 output" 문구.
3. **비가동 andon** (주황) — 정상일 때 "■ 비가동 시작 → 사유 선택", 비가동일 때 "▶ 가동 재개"(경과·사유 표시). 한 탭이 두 테이블 기록.
4. **지난 교대 마감** (주황-red, 조건부) — 마감대기 교대가 있을 때만. "최종 생산 [___] 개 · [마감]" (진척 있으면 prefill). 명시적 교대 귀속.
5. **불량 대기** (초록, 조건부) — 확정대기(불량) record가 있을 때만. "불량 [___] 개 · [확정]".

조건부 섹션(4·5)은 해당 대기가 없으면 숨긴다. 대기 건수는 목록/헤더 **배지**로 독촉.

### 4.3 라이프사이클

```
진행중 ──교대시계 종료──▶ 마감대기(무기한·배지) ──교대 마감──▶ 확정대기·불량(무기한·배지) ──불량 입력──▶ 확정
```

- **진행중**: `production_progress_reports`(진척) · `downtime_entries`/`machine_logs`(비가동). 실시간 가동×성능.
- **마감대기**: 교대 시계는 끝났으나 최종 수량 미확정. 자동으로 아무것도 닫지 않는다.
- **교대 마감**(수동): 최종 수량 확정 → `production_records` 생성(output O, **defect NULL**), 가동×성능 확정.
- **확정대기·불량**: output 확정, 품질/OEE 보류.
- **불량 입력**(다음날): 검사분 입력 → 품질·OEE 확정.

### 4.4 늦은 입력 & 귀속

- "교대 마감"은 자동 컷오프가 아니라 **수동·늦어도 되는** 동작. 진척을 입력해 뒀으면 마지막값이 마감칸에 prefill되어 원탭 확정, 종이값만 있으면 직접 입력.
- 21:40(B조 중)에 A조를 마감해도 **A조로 귀속**. 콘솔은 "현재 교대 진척"과 "지난 교대 마감"을 **병행** 표시. (Finding 3의 "모달 교대 스냅샷"과 같은 귀속 원칙의 확장 — props의 라이브 shift가 아니라 대상 교대를 명시적으로 지정.)
- 마감·불량 대기는 무기한. 배지로 독촉하되 강제 마감 없음.

### 4.5 비가동 andon (한 동작, 두 테이블)

작업자의 한 동작("비가동 시작+사유" / "가동 재개")이 상태(`machine_logs` 비정상 구간)와 비가동(`downtime_entries` 구간)을 **함께** 기록한다. 겹치는 두 구간은 OEE 계산의 유니온(이미 반영된 `calculateVerifiedDowntimeMinutesForWindow`)이 한 번만 센다 → 이중계산 없음. 사유는 사전 정의 코드 그리드에서 선택.

## 5. 데이터 모델 영향

### 유지 (안 깨짐)
- `production_records` / `production_progress_reports` / `downtime_entries` / `machine_logs` 테이블·용도 그대로.
- 실시간 기능(가동×성능·진척·경과율), OEE 계산 규율(per-piece tact, NULL≠0%, 확정 스냅샷) 그대로.

### 추가/보강 (additive)
- **`production_records.defect_qty` NULL 허용** = 미검사(≠불량0). OEE 계산이 "품질 NULL = 보류"를 이해(실시간 기능과 같은 규율). 저장 경로 5곳(oeeRules·route·daily·[recordId]·ShiftDataInputForm)의 검증에서 defect NULL을 허용/전파.
- **확정 상태 3단계**(마감대기 → 확정대기·불량 → 확정). 파생 우선: `production_shift_states`(WORKING) − 확정 `production_records`로 마감대기 백로그를, output O·defect NULL로 불량대기를 도출. 파생으로 부족하면 작은 상태 플래그를 추가(구현 단계 결정).
- **"교대 마감"** = `production_records` 생성·output 확정 시점(수동·늦어도 됨). output은 해당 교대의 마지막 진척값에서 파생.
- **운영자 콘솔이 상태변경·생산실적·진행보고를 대체** — ②의 생산실적 모달(중복), ③ 진행보고, 상태변경을 콘솔 섹션으로 흡수.

## 6. 진입점 정리

- **운영자**: 콘솔로 통합. `/operator-view`(현 `OperatorDashboard`) 진화.
- **관리자**: 데이터 입력 페이지 존치 — 과거/누락분 수정·일괄 보정용 오버라이드 경로.

## 7. i18n / 역할 / 반응형

- 새 문구 ko/vi 동시. 기존 `machines`/`production` 네임스페이스 재사용.
- 역할: operator 콘솔 입력. 관리자/엔지니어는 조회 + 데이터입력 페이지.
- 태블릿 가로 우선, 큰 터치 타깃. 데스크톱에서도 열람 가능해야 하나 밀도 최적화는 비목표.

## 8. 리스크 / 열린 질문 (구현 계획에서 해소)

- **defect NULL 도입의 파급**: 기존 리포트·집계·OEE 뷰가 "확정대기(품질 NULL)"를 확정과 구분해 표시/제외해야 함. 회귀 범위 점검 필요.
- **무기한 백로그 누적**: 마감·불량 대기가 쌓일 때 정렬·배지·오래된 건 강조 정책.
- **마감 prefill 부재**: 진척을 한 번도 안 넣은 교대는 마감칸이 비어 종이값 직접 입력.
- **확정 상태를 파생 vs 실제 컬럼**: 우선 파생, 성능/명확성 문제 시 플래그 — 구현 단계 판단.
- **데이터입력 페이지 defect 동시 입력**: 관리자 백필 시엔 output+defect 함께 넣을 수 있어야 하므로, defect NULL 허용과 "동시 입력 허용"이 공존해야 함(둘 다 유효 경로).

## 9. 비고

- 이 설계는 앞서 반영한 실시간 모니터링 기능(가동×성능·진행보고 RPC·서버 교대창·모달 교대 스냅샷) 위에 얹힌다. 특히 4.4의 귀속은 Finding 3과 동일 원칙.
- 구현·마이그레이션·main 병합은 사용자의 명시적 지시가 있을 때만. 이 문서는 설계까지.
